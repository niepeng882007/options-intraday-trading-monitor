## 一、变更摘要

项目原有的 `Dockerfile` 和 `docker-compose.yaml` 是为 macOS 本地开发设计的，存在多个在 Linux VPS 上无法正常运行的问题：`deploy` 段在非 Swarm 模式下无效、`host.docker.internal` 在 Linux 上不可用、无 `.dockerignore` 导致 `.env` 泄入镜像、healthcheck 仅检查 PID 无法反映真实服务状态、缺乏日志轮转和运维工具。

本次变更修复了上述全部问题，新增了 FutuOpenD systemd 服务管理、健康检查告警脚本、每日自动备份脚本，以及一份面向非运维用户的完整部署指南。最终实现：一键部署、开机自启、故障自愈告警、数据定期备份。

## 三、关键代码解析

### 3.1 docker-compose.yaml 的 Linux 适配

**改动前：**
```yaml
version: "3.8"

services:
  playbook:
    # ...
    deploy:               # ← 非 Swarm 模式下完全被忽略
      resources:
        limits:
          memory: 1G
          cpus: "1.0"
    healthcheck:
      test: ["CMD", "python", "-c", "import os; os.kill(1, 0)"]  # ← 只检查 PID 1 是否存在
```

**改动后：**
```yaml
services:                   # 移除 version 字段（Compose V2 已废弃）
  playbook:
    user: "${UID:-1000}:${GID:-1000}"          # 容器用户与宿主机 data/ 目录权限匹配
    extra_hosts:
      - "host.docker.internal:host-gateway"    # Linux 上启用 host.docker.internal
    mem_limit: 1g           # 替代 deploy.resources（非 Swarm 兼容）
    cpus: 1.0
    healthcheck:
      test: ["CMD", "python", "-c",
        "import socket; s=socket.create_connection(('host.docker.internal', 11111), 5); s.close()"]
      interval: 120s        # 给 FutuOpenD 短暂重启留恢复窗口
      retries: 5
    logging:
      driver: json-file
      options:
        max-size: "50m"     # 单文件最大 50MB
        max-file: "5"       # 最多保留 5 个文件 = 250MB 上限
```

**解析：**

这里有 5 个关键决策：

1. **`deploy` → `mem_limit`/`cpus`**：Docker Compose 的 `deploy` 段是为 Docker Swarm 设计的，在 `docker compose up` 模式下会被静默忽略，意味着原配置实际上没有资源限制。`mem_limit` 和 `cpus` 是 Compose V2 的非 Swarm 替代方案。

2. **`extra_hosts: host-gateway`**：macOS 上 Docker Desktop 自动注入 `host.docker.internal` DNS，但 Linux 上的 Docker Engine 不会。`host-gateway` 是 Docker 20.10+ 提供的魔术值，会被替换为宿主机的 bridge 网关 IP。

3. **`user: "${UID:-1000}:${GID:-1000}"`**：容器进程以指定 UID/GID 运行，确保写入 `./data` volume 的文件权限与宿主机用户一致，避免 root 写入后宿主机无法读取的问题。

4. **healthcheck 从 PID 改为 TCP 探测**：`os.kill(1, 0)` 只能检查容器主进程是否存在，即使应用已经无法连接 FutuOpenD 也会报告 healthy。改为探测 FutuOpenD 的 11111 端口，真正反映服务可用性。

5. **日志轮转**：Docker 默认 `json-file` 驱动无大小限制，长期运行会撑满磁盘。`max-size` + `max-file` 实现自动轮转。

### 3.2 .dockerignore 防止敏感信息泄露

```dockerignore
.git          # Git 历史（可达数百 MB）
.env          # ← 关键：包含 Telegram Bot Token
.env.*        # 所有 .env 变体
.claude       # Claude Code 配置
data/         # 运行时数据（SQLite、缓存）
tests/        # 测试代码无需进入生产镜像
venv/         # 本地虚拟环境
__pycache__   # Python 字节码缓存
*.pyc
*.md          # 文档
!requirements.txt  # ← 例外：requirements.txt 虽然是 .md 外的文件，但 *.md 规则不影响它；这里显式保留以防未来添加更多排除规则
docker-compose.yaml  # compose 文件本身不需要进镜像
```

**解析：**

没有 `.dockerignore` 时，`docker build` 会把整个项目目录作为 build context 发送给 Docker daemon。即使 `Dockerfile` 中没有 `COPY .env`，`.env` 文件仍然会进入 build context，某些情况下可被中间层缓存泄露。`.dockerignore` 从源头切断这个风险。

### 3.3 systemd 服务的重启策略

```ini
[Service]
Restart=on-failure        # 仅在非零退出时重启（正常 stop 不重启）
RestartSec=10             # 重启间隔 10 秒，避免疯狂重启
StartLimitIntervalSec=300 # 5 分钟内...
StartLimitBurst=5         # ...最多重启 5 次，超过则放弃（需人工介入）
```

**解析：**

这是 systemd 的两层防护机制：`Restart` + `RestartSec` 处理偶发崩溃，`StartLimit*` 防止持续崩溃导致无限重启消耗资源。如果 FutuOpenD 在 5 分钟内崩溃 5 次，systemd 会标记服务为 `failed`，此时 healthcheck.sh 会检测到端口不可达并发送 Telegram 告警。

### 3.4 healthcheck.sh 的多层检查

```bash
# 检查 1: FutuOpenD TCP 端口探测
if ! nc -z 127.0.0.1 "$FUTU_PORT" 2>/dev/null; then
    sudo systemctl restart futuopend  # 自动尝试恢复
    sleep 5
    if ! nc -z 127.0.0.1 "$FUTU_PORT" 2>/dev/null; then
        ALERTS="${ALERTS}🔴 FutuOpenD 端口不可达，重启后仍无法恢复\n"
    fi
fi

# 检查 2: Docker 容器运行状态
CONTAINER_STATUS=$(docker inspect --format='{{.State.Status}}' "$CONTAINER_NAME" 2>/dev/null)

# 检查 3: 磁盘使用率
DISK_USAGE=$(df / | awk 'NR==2 {gsub(/%/,""); print $5}')

# 检查 4: 内存可用百分比
MEM_FREE_PCT=$(free | awk '/^Mem:/ {printf "%.0f", $7/$2*100}')
```

**解析：**

脚本遵循"先尝试自愈，再告警"的模式。FutuOpenD 端口不可达时先自动 `systemctl restart`，等待 5 秒后二次检查，仅在自愈失败后才发送告警。这避免了因 FutuOpenD 短暂重启（如自动更新）导致的误报。

### 3.5 backup.sh 的 SQLite 安全备份

```bash
sqlite3 "$DB_FILE" ".backup '${BACKUP_PATH}/monitor.db'"
```

**解析：**

直接 `cp` SQLite 数据库文件在数据库有写入操作时可能得到损坏的副本（因为 WAL 日志未刷入）。SQLite 的 `.backup` 命令是原子操作，它会获取读锁、复制所有页、确保 WAL 一致性，即使在备份过程中有并发写入也能得到一致的数据库副本。

## 四、涉及的知识点

### 4.1 Docker Build Context 与镜像安全

**是什么：** `docker build` 时，Docker CLI 会将指定目录（build context）的所有文件打包发送给 Docker daemon。`.dockerignore` 文件控制哪些文件被排除在 build context 之外。

**为什么重要：** 即使 `Dockerfile` 中没有显式 `COPY` 敏感文件，它们仍然存在于 build context 中。在多阶段构建、缓存泄露、或误用 `COPY . .` 的场景下，敏感信息（API Key、Token）可能被嵌入镜像层。镜像一旦推送到 registry，秘密就永久暴露。

**在本次变更中如何体现：** `.dockerignore` 排除了 `.env`（含 Telegram Bot Token）和 `.env.*` 变体，从源头阻止敏感信息进入 build context。同时排除 `.git`（通常数百 MB）和 `data/`（运行时数据），显著减小 build context 体积和构建时间。

**延伸阅读：** 搜索 "Docker build context security best practices"、"Docker image layer secrets leaking"、"docker history --no-trunc"

### 4.2 Docker Compose V2 的 Swarm vs 非 Swarm 配置

**是什么：** Docker Compose 文件中的 `deploy` 段（`deploy.resources.limits`）是 Docker Swarm 模式的专属配置。在 `docker compose up`（非 Swarm）模式下，这些配置会被静默忽略，不会报错也不会生效。

**为什么重要：** 这是一个非常隐蔽的陷阱。开发者以为配置了内存限制 `memory: 1G`，但容器实际上没有任何资源限制，可能在内存泄露时吃掉整台 VPS 的内存导致 OOM Killer 介入。Docker Compose V2 提供了 `mem_limit`、`cpus` 等顶层属性作为非 Swarm 替代方案。

**在本次变更中如何体现：** 将 `deploy.resources.limits.memory: 1G` 替换为 `mem_limit: 1g`，将 `deploy.resources.limits.cpus: "1.0"` 替换为 `cpus: 1.0`，确保资源限制在 `docker compose up` 模式下真正生效。

**延伸阅读：** 搜索 "docker compose deploy vs non-swarm"、"docker compose v2 resource limits"、"docker compose mem_limit vs deploy resources"

### 4.3 Linux 上的 host.docker.internal 与容器-宿主机通信

**是什么：** `host.docker.internal` 是一个特殊 DNS 名称，用于让容器内的进程访问宿主机上的服务。在 macOS/Windows 的 Docker Desktop 中自动可用，但在 Linux 的 Docker Engine 中默认不存在。

**为什么重要：** 本项目的应用容器需要连接运行在宿主机上的 FutuOpenD（端口 11111）。如果使用 `network_mode: "host"` 虽然能解决问题，但会完全移除容器的网络隔离，容器共享宿主机的所有端口和网络接口，违反最小权限原则。

**在本次变更中如何体现：** 使用 `extra_hosts: ["host.docker.internal:host-gateway"]`。`host-gateway` 是 Docker 20.10+ 提供的魔术值，Docker daemon 会将其替换为宿主机的 bridge 网关 IP（通常是 `172.17.0.1`）。这样既保留了网络隔离，又统一了 macOS/Linux 的配置（不需要为不同平台维护不同的 compose 文件）。

**延伸阅读：** 搜索 "docker extra_hosts host-gateway"、"docker host.docker.internal linux"、"docker network_mode host vs bridge"

### 4.4 systemd 服务管理的两层重启防护

**是什么：** systemd 的服务重启机制分为两层：`Restart`/`RestartSec` 控制单次重启行为，`StartLimitIntervalSec`/`StartLimitBurst` 控制重启频率上限。两层配合实现"偶发故障自动恢复 + 持续故障及时止损"。

**为什么重要：** 只配置 `Restart=always` 而不设置频率限制，会导致持续崩溃的服务无限重启，消耗 CPU 和日志空间，甚至可能因频繁启动产生副作用（如重复发送消息、创建重复连接）。只配置频率限制而不配置 `RestartSec`，会导致服务在毫秒级内快速重启多次然后被 limit 永久禁止。

**在本次变更中如何体现：** FutuOpenD 配置为 `Restart=on-failure`（正常停止不重启）+ `RestartSec=10`（每次重启间隔 10 秒）+ 5 分钟内最多 5 次。如果 FutuOpenD 因 token 过期持续崩溃，50 秒内重启 5 次后 systemd 会标记为 failed 并停止重试，此时 healthcheck.sh 会检测到端口不可达并通过 Telegram 告知用户需要人工续期。

**延伸阅读：** 搜索 "systemd restart policy best practices"、"systemd StartLimitBurst"、"systemd service hardening"

## 五、测试建议

### 5.1 建议的测试用例

- **正常场景：**
  - `docker compose config` 解析无错误
  - `docker compose build` 构建成功
  - 构建的镜像中不包含 `.env`（`docker run --rm <image> cat /app/.env` 应失败）
  - 容器启动后 `data/` 目录可写
  - Telegram 发送 `SPY` 收到正常回复

- **边界场景：**
  - FutuOpenD 未启动时容器的 healthcheck 状态应为 unhealthy
  - `UID`/`GID` 环境变量未设置时默认使用 1000:1000
  - healthcheck.sh 在 Telegram 凭据为空时不崩溃（curl 静默失败）
  - backup.sh 在 `monitor.db` 不存在时跳过 SQLite 备份而非报错

- **异常场景：**
  - 模拟 FutuOpenD 崩溃：`sudo systemctl stop futuopend`，等待 healthcheck 告警
  - 模拟磁盘满：检查 healthcheck 是否在 90% 时告警
  - 模拟 VPS 重启：`sudo reboot` 后验证 FutuOpenD + 容器自动恢复

### 5.2 测试思路说明

部署类变更的测试重点不是代码逻辑正确性，而是**环境兼容性**和**故障恢复能力**。核心关注：(1) 开发环境（macOS）和生产环境（Linux）的行为一致性；(2) 各组件故障时的自愈和告警链路是否畅通；(3) 敏感信息是否被正确隔离。

## 六、场景扩展

### 6.1 类似场景

1. **任何 "本地开发 → 服务器部署" 的 Docker 项目**：`extra_hosts: host-gateway`、`.dockerignore`、`mem_limit` vs `deploy` 的区别是通用问题。几乎所有连接宿主机服务（Redis、PostgreSQL、其他 daemon）的容器化项目都会遇到。

2. **需要长期运行的后台服务（Trading Bot、爬虫、监控）**：systemd 的两层重启策略 + cron healthcheck + Telegram 告警是一套通用的轻量运维方案，适用于不值得上 Kubernetes 但需要稳定运行的单机服务。

3. **SQLite 数据库的生产备份**：任何使用 SQLite 作为持久存储的项目（小型 Web 应用、嵌入式系统、CLI 工具），都应该用 `.backup` 命令而非文件复制来备份，尤其是 WAL 模式下。

### 6.2 进阶思考

- **如果需要多台 VPS 部署同一套系统**（如美股和港股分别在不同地区的 VPS），当前的手动部署方式会变得难以维护。可以考虑引入 Ansible playbook 实现一键批量部署，或使用 Docker Swarm / K3s 进行轻量编排。

- **FutuOpenD 的 Token 续期是当前方案的最大人工干预点**。是否可以通过 Futu API 的 `unlock_trade` + `login_account` 接口实现自动续期？或者封装一个定时检测 token 有效性的脚本，在过期前 7 天主动提醒？

## 七、总结

本次变更的核心是将一个"能在本地跑"的 Docker 配置升级为"能在生产 VPS 上稳定运行"的部署方案。最关键的教训：**Docker Compose 的 `deploy` 段在非 Swarm 模式下静默失效**——这类"不报错但不生效"的配置陷阱是生产事故的常见根源。其次，macOS Docker Desktop 和 Linux Docker Engine 的行为差异（如 `host.docker.internal`）需要在 compose 文件中显式处理，而非依赖平台默认行为。`.dockerignore` 看似小事，但它是防止 API Token 泄入镜像的第一道防线。最后，systemd 两层重启 + cron healthcheck + Telegram 告警构成了一套简单但有效的"穷人运维"方案，适合不想上 K8s 的单机部署场景。
