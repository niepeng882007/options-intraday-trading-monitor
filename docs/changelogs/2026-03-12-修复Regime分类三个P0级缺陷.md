# 修复 US Playbook Regime 分类系统三个 P0 级缺陷

> 日期：2026-03-12
> 类型：Bug 修复
> 难度：⭐⭐⭐⭐

## 一、变更摘要

US Playbook 的 Regime 分类系统（`classify_us_regime()`）存在三个严重缺陷：**C1** — gap 百分比使用盘中实时价格计算而非开盘价，导致 gap 值随盘中波动持续漂移，GAP_AND_GO 与 TREND_DAY 分类在午后完全失真；**M5** — 市场广度（Breadth）的多空方向推断依赖脆弱的字符串解析，当空头占多数时初始值错误赋为 "bullish"，靠解析 `"3↑ 7↓ / 10"` 修正；**C3** — Market Tone 的 confidence modifier 在 auto-scan 路径完全缺失，且 pipeline 内部的 tone 调整代码是不可达的 dead code。

本次修复后：gap 计算锁定在开盘价（与 backtest 一致）、广度方向通过结构化字段传递（消除 string parsing）、auto-scan 和 on-demand 统一应用 tone modifier。

## 三、关键代码解析

### 3.1 C1: gap_pct 从"漂移量"变为"锁定量"

**改动前：**
```python
# regime.py:130 — 使用当前盘中价格计算 gap
gap_pct = ((price - prev_close) / prev_close * 100) if prev_close > 0 else 0.0
```

**改动后：**
```python
# regime.py:131-132 — 使用开盘价计算 gap，fallback 保证向后兼容
gap_ref = open_price if open_price > 0 else price  # 未传 open_price 时退化为旧行为
gap_pct = ((gap_ref - prev_close) / prev_close * 100) if prev_close > 0 else 0.0
```

**解析：**

Gap（跳空）是一个**开盘时刻就确定的事实**——它反映的是市场在隔夜/盘前对新信息的定价偏移。用当前价格计算 gap 会导致两个问题：

1. **漂移失真**：开盘跳空 +2% 的股票午后回落到平盘，gap_pct 变成 0%，原本应分类为 GAP_AND_GO 的被误判为 TREND_DAY 或 UNCLEAR
2. **反向幽灵 gap**：没有跳空但盘中涨 2% 的股票，gap_pct 虚高到 2%，影响 `small_gap` 判断

关键设计决策是用 `open_price if open_price > 0 else price` 做 fallback，而不是强制要求 open_price。这确保了：
- 所有现有测试无需改动 `open_price` 参数（默认 0.0 → fallback 到旧行为）
- 新代码只要传了 open_price 就自动获得正确行为
- backtest 中 evaluators.py 已经有 `open_price` 变量，只需传入即可

同时 `market_tone.py` 的 `_classify_gap()` 也有同样问题（用 `last_price` 给 SPY 算 gap），同步修复。

### 3.2 M5: 用结构化字段替代字符串解析

**改动前：**
```python
# market_tone.py — _aggregate() 中的 breadth 方向推断
if breadth.alignment_label == "strong_aligned":
    majority_dir = (
        "bullish" if breadth.aligned_count > breadth.total_count / 2
        else "bearish"
    )
    # 字符串解析修正方向
    if "↑" in breadth.details:
        up_count = int(breadth.details.split("↑")[0])
        down_count = int(breadth.details.split("↑")[1].split("↓")[0].strip())
        majority_dir = "bullish" if up_count > down_count else "bearish"
    direction_votes.append(majority_dir)
```

**改动后：**
```python
# __init__.py — BreadthProxy 增加结构化字段
@dataclass
class BreadthProxy:
    aligned_count: int
    total_count: int
    alignment_ratio: float
    alignment_label: str
    index_aligned: bool
    majority_direction: str = "neutral"  # 新增：直接携带方向信息
    details: str = ""

# market_tone.py — _compute_breadth() 在计算时就设好方向
majority_dir = "bullish" if bullish_count >= bearish_count else "bearish"
return BreadthProxy(
    ...,
    majority_direction=majority_dir,  # 源头设置，而非消费端猜测
    details=f"{bullish_count}↑ {bearish_count}↓ / {total}",
)

# market_tone.py — _aggregate() 直接使用字段
if breadth.alignment_label == "strong_aligned":
    direction_votes.append(breadth.majority_direction)  # 干净、不可能出错
```

**解析：**

原代码的 bug 链条：
1. `aligned_count = max(bullish, bearish)` — 只保留了数量，**丢失了方向信息**
2. `aligned_count > total_count / 2` — 这个条件判断的是"是否有多数方"，不是"多数方是哪边"
3. 当 bear=7, bull=3 时，`aligned_count=7 > 10/2=5` → True → `majority_dir = "bullish"` — **方向反了！**
4. 然后靠解析 `"3↑ 7↓ / 10"` 字符串修正回 "bearish"

修复策略是**数据产生时就携带完整语义**（Producer Sets, Consumer Gets）：`_compute_breadth()` 在 line 392 已经算好了 `majority_dir`，只是没存到数据结构里。把它放进 `BreadthProxy` 后，消费端不需要做任何推断。

### 3.3 C3: 统一 tone modifier 应用路径

**改动前的问题地图：**
```
_run_analysis_pipeline(market_tone=None)  ← 所有调用者都不传此参数
    ├── lines 300-306: if market_tone ...  ← DEAD CODE（永远为 False）
    │
generate_playbook_for_symbol()            ← on-demand 路径
    ├── tone = _ensure_market_tone()
    └── regime.confidence += tone.modifier ← OK，但逻辑与 dead code 重复
    │
_l2_verify()                              ← auto-scan 路径
    └── （完全没有 tone 调整）            ← BUG：D 级市场信号 confidence 不降低
```

**改动后：**
```python
# main.py — 提取静态方法，统一两条路径
@staticmethod
def _apply_tone_modifier(regime: USRegimeResult, tone: MarketTone) -> None:
    """Apply market tone confidence modifier to a regime result."""
    if tone.confidence_modifier == 0:
        return
    old_conf = regime.confidence
    regime.confidence = max(0.0, min(1.0, regime.confidence + tone.confidence_modifier))
    if abs(regime.confidence - old_conf) > 0.001:
        tone_note = f"Tone {tone.grade} adj {tone.confidence_modifier:+.2f}"
        regime.details = f"{regime.details}; {tone_note}" if regime.details else tone_note

# on-demand 路径
tone = await self._ensure_market_tone()
if tone and result.market_tone is None:
    result.market_tone = tone
    self._apply_tone_modifier(result.regime, tone)  # 统一调用

# auto-scan _l2_verify 路径 — 新增
tone = await self._ensure_market_tone()
if tone:
    if result.market_tone is None:
        result.market_tone = tone
    self._apply_tone_modifier(result.regime, tone)  # 与 on-demand 对称
```

**解析：**

这个修复涉及三步：

1. **删除 dead code**：`_run_analysis_pipeline` 的 `market_tone` 参数从未被传入，lines 300-306 永远不执行。删除它消除了"看起来在工作实际没工作"的误导。
2. **提取公共方法**：`_apply_tone_modifier` 是一个纯函数（给定 regime 和 tone，确定性地修改 confidence），提取为 `@staticmethod` 便于两条路径复用，也方便单元测试。
3. **补齐 auto-scan 路径**：在 `_l2_verify` 返回前（结构校验之前）应用 tone modifier，这样 D 级市场的 confidence 降低后，可能不满足 `bo_min_conf` 等门槛而被 L2 拒绝。

## 四、涉及的知识点

### 4.1 数据语义完整性（Semantic Integrity of Data）

**是什么：** 数据结构应当在产生时就包含完整的语义信息，消费者不应该需要"逆向推导"才能使用数据。

**为什么重要：** M5 bug 的根因就是 `BreadthProxy` 保存了 `aligned_count`（数量）却丢失了"aligned to which direction"（方向）。消费端被迫从 `details` 字符串逆向解析方向——这种"数据不自描述"的设计是 silent failure 的温床。字符串格式一旦变化（比如改用 `7↑3↓` 去掉空格），解析就会 crash 或返回错误结果，而且没有编译器/类型检查能发现。

**在本次变更中如何体现：** 给 `BreadthProxy` 增加 `majority_direction` 字段，在 `_compute_breadth()` 产生数据时就写入方向，消费端直接读取。代码从 12 行缩减到 1 行，且不可能出错。

**延伸阅读：** "Tell, Don't Ask" 原则、Rich Domain Model vs Anemic Domain Model、dataclass 设计最佳实践。

### 4.2 不变量（Invariant）与时变量（Time-Varying）的区分

**是什么：** 不变量是在某个时间点确定后不再改变的值；时变量是随时间持续变化的值。混淆两者会导致逻辑漂移。

**为什么重要：** C1 bug 的本质就是把 gap（不变量——开盘时刻确定）用时变量（当前价格）来计算。这导致同一个 regime 分类函数在 09:35 和 14:00 对同一只股票给出不同的 gap_pct，进而可能翻转 regime 判定。在量化系统中，这种"输入漂移"特别危险——backtest 用开盘价算 gap（正确），production 用实时价（错误），导致 backtest 结果不可复现。

**在本次变更中如何体现：** 将 `gap_ref` 固定为 `open_price`，与 backtest 对齐。同时保留 fallback（`open_price=0` 时用 `price`），避免破坏未传参的旧调用路径。

**延伸阅读：** 搜索 "event time vs processing time"、"snapshot isolation"、"backtest-production parity"。

### 4.3 Dead Code 的危害与识别

**是什么：** Dead code 是永远不会被执行到的代码。它不是"无害的注释"——它会误导维护者认为某个功能已经实现。

**为什么重要：** C3 中的 `market_tone` 参数和 lines 300-306 看起来在 `_run_analysis_pipeline` 内部处理了 tone 调整，但实际上没有任何调用者传入 `market_tone`。维护者（或代码审查者）看到这段代码会认为"tone 已经在 pipeline 里处理了"，从而不会去检查 auto-scan 路径是否也有 tone 调整——而实际上 auto-scan 完全没有。

**在本次变更中如何体现：**
1. 删除了不可达参数和代码块
2. 将真正的 tone 逻辑提取为 `_apply_tone_modifier()` 静态方法
3. 在两条路径（on-demand / auto-scan）显式调用，消除歧义

**延伸阅读：** "Dead code elimination"、SonarQube/Pyright 静态分析、"code as documentation" 理念。

### 4.4 Fallback 设计与向后兼容

**是什么：** 在修改函数签名时，通过默认值和 fallback 逻辑确保未更新的调用者不会 break。

**为什么重要：** `classify_us_regime` 有 20+ 处调用（production + test），硬性要求所有调用者立即传入 `open_price` 会导致大面积改动和高回归风险。

**在本次变更中如何体现：**
```python
open_price: float = 0.0  # 默认值 = 不传
gap_ref = open_price if open_price > 0 else price  # fallback 到旧行为
```
这让 20+ 个测试用例无需任何改动就继续通过，同时新代码只要传了 `open_price` 就获得正确行为。这是**渐进式迁移**（Gradual Migration）的典型手法。

**延伸阅读：** "Expand-Contract pattern"、API 版本兼容策略、Python 函数签名设计最佳实践。

### 5.2 测试思路说明

本次测试重点是**边界值和路径覆盖**：
- C1 的核心是"两个不同的输入（open vs current）是否产生不同的结果" → 需要对比测试
- M5 的核心是"方向信息是否正确传递" → 需要构造 bearish 多数场景（容易被 bullish default 掩盖）
- C3 的核心是"两条路径是否行为一致" → 需要验证 modifier 的算术正确性和边界夹紧

## 六、场景扩展

### 6.1 类似场景

1. **K 线 OHLC 字段混用**：很多量化系统在计算涨跌幅时混用 Open/Close/Last，导致日内信号不稳定。排查方法：对每个用到 `price` 的地方，明确它代表的是 Open、Close、Last、还是 VWAP。

2. **数据结构信息丢失**：类似 M5 的问题在日志系统中很常见——把结构化数据序列化为 string，然后在另一处用正则提取。更好的做法是始终保持结构化（dict/dataclass/protobuf），只在最终展示层做格式化。

3. **多路径行为不一致**：C3 的 on-demand vs auto-scan 不一致，在微服务系统中对应"API 路径 vs 消息队列路径"处理逻辑不一致。解决方案是提取公共业务逻辑层，两条路径都调用同一个方法。

### 6.2 进阶思考

1. **Backtest-Production Parity（回测-生产一致性）**：C1 修复后 production 的 gap 计算与 backtest 对齐了。但还有没有其他"backtest 用一套逻辑、production 用另一套"的地方？一个系统性的做法是让 production 和 backtest 共享同一个 `classify_us_regime()` 调用，确保参数来源一致。

2. **静态分析作为安全网**：C3 的 dead code 如果有 CI 集成 Pyright/mypy 严格模式，`market_tone` 参数未使用的 warning 可以早期发现。考虑在 CI 中增加 `--warn-unused-parameter` 类的检查。

3. **Tone Modifier 的 Regime 差异化（M4 延伸）**：当前 tone modifier 对所有 regime 一视同仁。逻辑上，D 级（choppy）市场应该 boost FADE_CHOP（mean reversion 策略更适合 chop），而 penalize GAP_AND_GO。这需要将 `_apply_tone_modifier` 从简单加减改为查表 `{(grade, regime): modifier}`。

## 七、总结

本次修复了 Regime 分类系统的三个 P0 级缺陷。最核心的教训是：**不变量和时变量不能混用**（C1）、**数据结构要自描述**（M5）、**dead code 比没有代码更危险**（C3）。修复策略上，通过 fallback 默认值实现了渐进式迁移（20+ 个现有测试无需改动），通过提取公共方法确保了 on-demand 和 auto-scan 两条路径的行为一致性。254 个测试全部通过，验证了修复的正确性和向后兼容性。
