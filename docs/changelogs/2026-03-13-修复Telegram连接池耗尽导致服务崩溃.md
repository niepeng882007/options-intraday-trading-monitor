# 修复 Telegram 连接池耗尽导致服务启动崩溃

> 日期：2026-03-13
> 类型：Bug 修复
> 难度：⭐⭐

## 一、变更摘要

服务在启动阶段反复崩溃，Telegram bot 无法完成初始化，报错 `Pool timeout: All connections in the connection pool are occupied`。根因是 `python-telegram-bot` 底层使用的 `httpx.AsyncClient` 默认连接池大小仅为 1，在通过代理（HTTPS_PROXY）访问 Telegram API 时，单连接极易被长时间占用，导致后续请求排队超时。

修复方案：将 `HTTPXRequest` 的 `connection_pool_size` 从默认值 1 增大到 8，同时将 `pool_timeout` 从 5 秒放宽到 10 秒，确保启动阶段和并发消息处理场景下有足够的连接余量。

## 二、修改的文件清单

| 文件路径 | 变更类型 | 说明 |
|---------|---------|------|
| `src/main.py` | 修改 | 增加 `TELEGRAM_CONNECTION_POOL_SIZE=8` 常量，`pool_timeout` 从 5s 调整为 10s，两个 `HTTPXRequest` 实例均传入 `connection_pool_size` |

## 三、关键代码解析

### 3.1 新增连接池大小常量

**改动前：**
```python
TELEGRAM_POOL_TIMEOUT_SECONDS = 5
```

**改动后：**
```python
TELEGRAM_POOL_TIMEOUT_SECONDS = 10
TELEGRAM_CONNECTION_POOL_SIZE = 8
```

**解析：**
`pool_timeout` 是等待连接池中空闲连接的超时时间。原来 5 秒配合 1 个连接，意味着如果那唯一的连接正在被 `get_me()` 或 `getUpdates` 长轮询占用，新请求只能等 5 秒就放弃。通过代理的连接建立本身就慢（TCP → TLS → proxy handshake），5 秒很可能不够。

`connection_pool_size=8` 是关键修复：允许同时保持 8 个到 Telegram API 的 TCP 连接。这样即使 `getUpdates` 长轮询占用了一个连接，其他 7 个连接仍可用于发送消息、初始化等操作。

### 3.2 两个 HTTPXRequest 实例都需要设置

**改动后：**
```python
# 用于发送消息的 request
request = HTTPXRequest(
    read_timeout=TELEGRAM_READ_TIMEOUT_SECONDS,
    write_timeout=TELEGRAM_WRITE_TIMEOUT_SECONDS,
    connect_timeout=TELEGRAM_CONNECT_TIMEOUT_SECONDS,
    pool_timeout=TELEGRAM_POOL_TIMEOUT_SECONDS,
    connection_pool_size=TELEGRAM_CONNECTION_POOL_SIZE,  # 新增
    proxy=proxy_url,
)
# 专用于 getUpdates 长轮询的 request
get_updates_request = HTTPXRequest(
    read_timeout=TELEGRAM_READ_TIMEOUT_SECONDS,
    write_timeout=TELEGRAM_WRITE_TIMEOUT_SECONDS,
    connect_timeout=TELEGRAM_CONNECT_TIMEOUT_SECONDS,
    pool_timeout=TELEGRAM_POOL_TIMEOUT_SECONDS,
    connection_pool_size=TELEGRAM_CONNECTION_POOL_SIZE,  # 新增
    proxy=proxy_url,
)
```

**解析：**
`python-telegram-bot` 的 `Application` 使用两个独立的 HTTP 客户端：
- **`request`**：发送消息、`get_me()`、设置 commands 等常规 API 调用
- **`get_updates_request`**：专门用于 `getUpdates` 长轮询（blocking call，通常 30 秒超时）

这是 `python-telegram-bot` 的设计优化 — 将长轮询隔离到单独的连接池，避免长轮询阻塞消息发送。但如果两个池都只有 1 个连接，初始化阶段（`get_me()` + `setMyCommands()` 连续调用）仍然会互相排队。

## 四、涉及的知识点

### 4.1 HTTP 连接池（Connection Pool）

**是什么：** HTTP/1.1 的 Keep-Alive 机制允许在同一个 TCP 连接上发送多个请求。连接池管理一组预建立的连接，避免每次请求都重新握手（TCP + TLS）。`connection_pool_size` 决定池中最大连接数。

**为什么重要：** 在代理环境下，每次建立新连接需要经过 `客户端 → 代理 → Telegram API` 的完整握手链路，延迟显著增加。如果池中只有 1 个连接，所有请求变成串行，任何一个慢请求都会阻塞后续请求，最终触发 `pool_timeout`。

**在本次变更中如何体现：** 服务启动时需要连续调用 `get_me()` 和 `setMyCommands()`，同时 `getUpdates` 长轮询也在抢连接。1 个连接根本不够用，增大到 8 后，这些请求可以并行执行。

**延伸阅读：** 搜索 `httpx connection pool size`、`httpcore PoolTimeout`、`HTTP Keep-Alive vs Connection Pooling`。

### 4.2 python-telegram-bot 的双 Request 架构

**是什么：** `python-telegram-bot` v20+ 使用 `Application.builder()` 模式构建 bot，支持分别配置 `request`（常规 API 调用）和 `get_updates_request`（长轮询专用）两个 HTTP 客户端。

**为什么重要：** `getUpdates` 是一个长轮询（long polling）调用，默认 timeout 约 30 秒。如果它和消息发送共享同一个连接池，消息发送可能被长轮询阻塞。双 Request 架构通过隔离连接池解决了这个问题。

**在本次变更中如何体现：** 两个 `HTTPXRequest` 实例都需要设置 `connection_pool_size`，否则即使常规请求的池扩大了，`getUpdates` 的池仍然可能成为瓶颈。

**延伸阅读：** 搜索 `python-telegram-bot HTTPXRequest`、`telegram bot getUpdates long polling`、`Application.builder request vs get_updates_request`。

### 4.3 代理环境下的超时调优

**是什么：** 通过 HTTP/SOCKS 代理访问外部 API 时，请求链路变长（多一跳），连接建立和数据传输都会变慢。需要相应调大各类超时参数。

**为什么重要：** 默认超时参数通常假设直连场景。代理环境下不调整，会导致间歇性超时，且难以复现（取决于代理负载和网络抖动）。

**在本次变更中如何体现：** `pool_timeout` 从 5s 调到 10s，给代理连接更多等待时间。但更关键的是增大池容量 — 超时只是治标，池太小才是根因。

**延伸阅读：** 搜索 `httpx proxy timeout tuning`、`HTTPS_PROXY connection latency`。

## 五、测试建议

### 5.1 建议的测试用例

- **正常场景：** 服务启动后，通过 Telegram 发送港股代码（如 `09988`）和美股代码（如 `AAPL`），确认都能正常返回 playbook
- **并发场景：** 快速连续发送 3-5 个查询消息，确认不会出现 PoolTimeout
- **冷启动场景：** 完全停止服务后重新启动，确认 `Telegram bot started` 日志正常出现，不再有 bootstrap retry
- **代理断开场景：** 临时断开代理，确认服务不会 crash，而是走 retry 逻辑并在代理恢复后自动重连

### 5.2 测试思路说明

核心关注点是**连接池耗尽**是否还会发生。最有效的验证方式是观察服务启动日志 — 如果看到 `Telegram bot started` 而没有 `bootstrap failed` 警告，说明修复生效。并发场景的测试则验证运行时的连接池够不够用。

## 六、场景扩展

### 6.1 类似场景

1. **任何通过代理访问外部 API 的服务**：如果你的服务需要通过公司代理访问 OpenAI API、Slack API 等，同样需要关注 HTTP 客户端的连接池大小和超时设置。默认值几乎都偏小。

2. **微服务间的 HTTP 调用**：服务 A 调用服务 B，如果连接池不够大，高并发时同样会 PoolTimeout。常见于 Python 的 `aiohttp`、`httpx`，以及 Go 的 `http.Client`（默认 `MaxIdleConnsPerHost=2`）。

3. **数据库连接池**：概念完全一致 — SQLAlchemy 的 `pool_size`、Redis 的 `max_connections`、PostgreSQL 的 `max_connections` 都是同一类问题。池太小会导致请求排队超时。

### 6.2 进阶思考

- **连接池大小如何确定？** 8 是一个经验值。理想的大小 = 最大并发请求数。对于 Telegram bot，可以估算：1 个 getUpdates + 同时处理的最大消息数（watchlist 里 20 个 symbol 同时查询？不太可能，通常 2-3 个就够了）。8 留了足够的余量。

- **是否需要连接池监控？** 生产环境中，可以通过 `httpx` 的 event hooks 或自定义中间件监控连接池使用率。如果池使用率长期 > 80%，说明需要扩容。

## 七、总结

这次修复的核心教训是：**HTTP 客户端的默认连接池配置几乎都偏保守**，尤其在代理环境下。`httpx` 默认 `connection_pool_size=1`，对于需要并发发送消息的 Telegram bot 来说完全不够用。

排查过程的关键一步是读日志 — 错误信息 `Pool timeout: All connections in the connection pool are occupied` 已经把根因说得很清楚了。修复只需要两行代码（增大池大小 + 放宽等待时间），但找到问题需要理解 httpx 的连接池模型和 python-telegram-bot 的双 Request 架构。

记住这个模式：**当你看到间歇性超时，尤其是在启动阶段或并发场景，先检查连接池配置**。
